## 先验知识

### 属性驱动设计（ADD）3.0 方法

ADD 方法是一种通过迭代精化进行软件架构设计的系统方法。每轮迭代遵循以下步骤：

**Step 1 - 审查输入**：审查输入内容，确定哪些需求将被视为架构驱动因素。输入包括：设计目的、主要功能（用例）、质量属性、架构关注点和约束。

**Step 2 - 通过选择驱动因素建立迭代目标**：一轮设计通常采用一系列设计迭代的形式，每轮迭代聚焦于实现一个特定目标。这个目标通常涉及设计以满足驱动因素的一个子集。

**Step 3 - 选择系统中一个或多个要精化的元素**：这一步是核心设计活动的开始。你选择的元素是与满足特定驱动因素相关的元素。对于全新开发（greenfield），你可以从建立系统上下文开始，然后选择唯一可用的元素——即系统本身——通过分解进行精化。对于现有系统或绿地系统的后续设计迭代，你通常会选择在先前迭代中确定的元素进行精化。

**Step 4 - 选择一个或多个满足所选驱动因素的设计概念**：这一步要求你识别可用于实现迭代目标的设计概念之间的替代方案，并从中选择一个。

**Step 5 - 实例化架构元素、分配职责并定义接口**：这一步要求根据所选设计概念实例化架构元素，并将职责分配给它们。此外，需要建立元素之间的关系和接口，使它们能够有效地协作和交换信息。

**Step 6 - 绘制视图并记录设计决策**：此时你已完成本轮迭代的设计活动。除了保存视图（你所创建的结构的表示）之外，还应记录设计迭代中做出的重要决策，以及这些决策背后的原因（即理由），以便后续分析和理解。

**Step 7 - 分析当前设计并审查迭代目标和设计目的的达成情况**：这一步检查是否已创建满足当前迭代目标的部分设计，并考虑是否需要额外的设计迭代。

### 案例研究：酒店定价系统（Hotel Pricing System）

**设计目的**：该项目可视为全新开发（greenfield development），因为它涉及对现有系统的完全替换。设计活动的目的是做出初始决策，以支持从头构建系统。

**主要功能**：

| 用例 | 描述 |
|------|------|
| HPS-1: 登录 | 用户（商业用户或管理员）在登录窗口提供凭据。系统根据用户身份服务检查这些凭据，如果成功则提供对系统的访问。登录后，用户只能对其被授权的酒店进行查询和更改。 |
| HPS-2: 更改价格 | 用户选择其被授权更改价格的特定酒店，并选择要对基础费率或固定费率进行价格更改的日期。所有从基础费率计算得出的费率价格在此时计算。系统允许在实际更改之前模拟价格更改。价格更改后，它们被推送到渠道管理系统（Channel Management System），并可供外部系统查询。 |
| HPS-3: 查询价格 | 用户或外部系统通过用户界面或查询 API 查询给定酒店的价格。 |
| HPS-4: 管理酒店 | 管理员添加、更改或修改酒店信息。这包括编辑酒店的税率、可用费率和房间类型。 |
| HPS-5: 管理费率 | 管理员添加、更改或修改费率。这包括定义不同费率的计算业务规则。 |
| HPS-6: 管理用户 | 管理员更改给定用户的权限。 |

**质量属性**：

| ID | 质量属性 | 场景 | 关联用例 | 对客户的重要性 | 实现难度 |
|----|---------|------|---------|-------------|---------|
| QA-1 | 性能（Performance） | 在正常运行期间，特定酒店和日期的基础费率价格被更改；该酒店所有费率和房间类型的价格在 100 毫秒内发布（可供查询）。 | HPS-2 | 高 | 高 |
| QA-2 | 可靠性（Reliability） | 用户对给定酒店执行多次价格更改；100% 的价格更改成功发布（可供查询），并且也被渠道管理系统接收。 | HPS-2 | 高 | 高 |
| QA-3 | 可用性（Availability） | 价格查询的正常运行时间 SLA 必须在维护窗口之外达到 99.9%。 | 全部 | 高 | 高 |
| QA-4 | 可伸缩性（Scalability） | 系统最初将通过其 API 支持每天至少 100,000 次价格查询，并且应能够处理高达 1,000,000 次而不使平均延迟增加超过 20%。 | HPS-3 | 高 | 高 |
| QA-5 | 安全性（Security） | 用户通过前端登录系统。用户的凭据根据用户身份服务进行验证，登录后只向其展示被授权使用的功能。 | 全部 | 高 | 中 |
| QA-6 | 可修改性（Modifiability） | 向系统添加支持不同于 REST 的协议（如 gRPC）的价格查询端点。新端点不需要对系统核心组件进行更改。 | 全部 | 中 | 中 |
| QA-7 | 可部署性（Deployability） | 应用程序作为开发过程的一部分在非生产环境之间移动。不需要修改代码。 | 全部 | 中 | 中 |
| QA-8 | 可监控性（Monitorability） | 系统运维人员希望在运行期间测量价格发布的性能和可靠性。系统提供一种机制，允许根据需要收集 100% 的这些指标。 | HPS-2 | 中 | 中 |
| QA-9 | 可测试性（Testability） | 系统及其所有元素的 100% 应支持独立于外部系统的集成测试。 | 全部 | 中 | 中 |

**架构关注点**：

| ID | 关注点 |
|----|-------|
| CRN-1 | 建立整体初始系统结构。 |
| CRN-2 | 利用团队关于 Java 技术、Angular 框架和 Kafka 的知识。 |
| CRN-3 | 将工作分配给开发团队成员。 |
| CRN-4 | 避免引入技术债务。 |
| CRN-5 | 建立持续部署基础设施。 |

**约束**：

| ID | 约束 |
|----|------|
| CON-1 | 系统后端必须使用 Java 开发。 |
| CON-2 | 前端必须使用 Angular 框架。 |
| CON-3 | 系统必须使用 Kafka 进行异步消息传递。 |
| CON-4 | 系统必须暴露 REST API 用于价格查询。 |
| CON-5 | 系统必须与外部用户身份服务（User Identity Service）集成进行认证。 |
| CON-6 | 系统必须与外部渠道管理系统（Channel Management System）集成以发布价格。 |
